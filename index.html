The vision

Companies run on thousands of processes. Most are not documented. That's fine.

But a few of these flows are critical.

Many companies struggle to define their mission critical flows--let alone execute them.

Examples of critical flows

Core user journeys
Ops procedures
System architecture
Integrations
Marketing & sales playbooks

Wait, aren't there already tools for this?

Quite a few! But they all ask the user to translate logic into visual structure to create a flowchart.

The effort required to create and keep flowcharts current means they are not used consistently.

In Splotch, all you need is text

We translate meaning into visual structure. And back again.

To create (or edit), just describe what you want.

Text and visuals are always in sync.

The big question is... what kinds of critical flows most need automated mapping?

candidate #1

understand the current state

  • Persona: You're a new hire; you joined a new team; you're starting a new project.
  • Task: You're doing a CUJ, ops or architecture audit. You need to figure out how things work now so you have context for the project.
  • Problem: tldr: you gotta read (a lot) of docs (if they exist at all). Once you do get a handle on how a process works, it stays in your head. The next person has to repeat this.

idea: map the docs

Transform existing documentation in Jira/Github/Linear/docs into birds-eye-view flows that update as the docs change. If docs don't exist, create a map by typing up what you learn. Add or edit as your understanding evolves, and the map evolves too--for everyone.

candidate #2

co-create the future state

  • Persona: You're a PM, engineer, designer, ops, sales or marketing manager.
  • Task: Design a new feature, campaign or playbook.
  • Problem: PRDs, marketing plans and ops manuals are hard to parse quickly and often miss edge cases. When colleagues weigh in, you spend most of the time bogged down in marginal details because they can't see a full picture of the desired future state.

idea: map the future state

Start with a text description of user stories and key milestones. Add edits by text as you uncover requirements and edge cases. Circulate to your team for confirmation. Now you have a reference that gives context to discussions.

candidate #3

collaborate from a shared map

  • Persona: You're a PM, engineer, designer, ops, sales or marketing manager.
  • Task: You need to get buy-in on a project that is plagued with ambiguities.
  • Problem: You're in meetings all day, but you're not getting closer to a shared picture of how things should work.

idea: compare diffs

Github lets engineers propose, discuss, revert, rework and finalize code by showing the differences between proposed solutions.

Splotch could help all teams do the same: ask for comments in context; turn comments into updated flows; identify where decisions need to be made; formalize decisions in a new flow.
